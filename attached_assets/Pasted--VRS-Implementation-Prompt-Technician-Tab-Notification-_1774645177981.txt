## TASK 1: Add "Technicians" Sub-Tab to User Management

### What to build
Within the Admin Dashboard's **User Management** section (`activeView === "users"`), add a two-tab layout at the top of the content area:
- **Tab 1: "Staff"** — The existing user management table, unchanged in functionality, but now filtered to EXCLUDE users where `role = 'technician'`
- **Tab 2: "Field Technicians"** — A new read-only table showing technicians who have logged in at least once

The Staff tab should be the default active tab. Tab switching should not cause a page reload — just swap which table is displayed.

### Backend: New API Endpoint

Create a new endpoint: `GET /api/admin/technician-users`

- Protected: `authenticateToken, requireRole("admin")`
- This endpoint queries technicians who have logged into the app by:
  1. Selecting from the `users` table WHERE `role = 'technician'` AND `isSystemAccount = false`
  2. For each technician user found, JOIN or cross-reference the `technicians` table (match on `users.racId = technicians.ldapId`) to get: `district`, `techUnNo`, `managerName`
  3. For each technician, compute ticket counts from the `submissions` table by counting rows where `submissions.technicianLdapId = users.racId`, grouped by `ticketStatus`:
     - `totalTickets` — count of all submissions
     - `pendingCount` — count where `ticketStatus IN ('queued', 'pending')`
     - `approvedCount` — count where `ticketStatus = 'completed'`
     - `rejectedCount` — count where `ticketStatus = 'rejected'`
  4. Return the combined data as a JSON array

**Do this in a SINGLE efficient SQL query using LEFT JOINs and conditional aggregation, not in a loop.**

Example response shape:
```json
{
  "technicians": [
    {
      "id": 45,
      "name": "ERNESTO GOMEZ",
      "racId": "egomez2",
      "phone": "7605808909",
      "district": "7252",
      "techUnNo": "0087252061",
      "managerName": "SMITH, JOHN",
      "totalTickets": 3,
      "pendingCount": 1,
      "approvedCount": 2,
      "rejectedCount": 0
    }
  ]
}
```

### Backend: Modify Existing Users Endpoint

This is the ONLY change to the existing endpoint. Do not modify `getUsers()` in storage.ts.

## TASK 2: Reduce Notification Sound Volume

### Problem
The notification ding in `client/src/lib/websocket.ts` is too loud for agents wearing headsets. It uses two simultaneous playback methods (WAV Audio element AND Web Audio API oscillators) that stack on top of each other, making the effective volume much louder than intended.

### Fix — In `client/src/lib/websocket.ts`:

**Change 1 — WAV Audio element volume:**
Find where the Audio element's volume is set (currently `0.5`). Change it to `0.15`.

**Change 2 — Web Audio API gain:**
In the `playTone()` function, find:
```
gain.gain.setValueAtTime(0.3, ctx.currentTime);
```
Change `0.3` to `0.08`.

**Change 3 — WAV buffer generation amplitude:**
In the `NOTIFICATION_SOUND_DATA` generation block, find the two amplitude multipliers (both currently `* 0.3`):
```
let sample = Math.sin(2 * Math.PI * freq1 * t) * envelope * 0.3;
```
and:
```
sample += Math.sin(2 * Math.PI * freq2 * t) * Math.exp(-(t - 0.15) * 8) * 0.3;
```
Change both `0.3` to `0.08`.

**These three changes together bring the notification to a subtle, professional-level ding that's audible but not jarring on headsets. Do not change the frequencies, duration, decay curve, or any other audio parameters.**
