REPLIT PROMPT — Add Client-Side Upload Diagnostics Logging
GOAL: Add a lightweight client-side error reporting endpoint so that when a technician's photo upload fails, we capture exactly what went wrong on the device. Currently, upload failures that happen client-side leave zero trace in server logs — we have no way to diagnose what happened. This adds a simple POST endpoint that the client calls when an upload fails, logging the error details server-side.
CRITICAL: Do NOT modify any existing upload behavior, UI, flow, file filtering, retry logic, or success paths. The only changes are: (1) a new server endpoint that receives error reports, and (2) the existing upload failure code paths now also fire a non-blocking report to that endpoint before continuing exactly as they do today.

Step 1 — New server endpoint: POST /api/uploads/report-error
Add a new route in server/replit_integrations/object_storage/routes.ts inside the existing registerObjectStorageRoutes function. Place it after the existing request-url route.

Route: POST /api/uploads/report-error
Middleware: authenticateToken only (no role restriction — we want this to work even if role check is the problem)
Request body expects JSON with these fields (all optional, log whatever arrives):

stage (string) — where in the flow it failed: "file-filter", "request-url", "xhr-upload", "xhr-timeout", "xhr-error", "unknown"
fileName (string)
fileSize (number)
fileType (string) — the raw file.type value from the browser
errorMessage (string)
xhrStatus (number | null)
attempt (number) — which retry attempt (0, 1, 2)
userAgent (string) — navigator.userAgent
timestamp (string) — ISO timestamp from the client


Handler: Just console.error("[UPLOAD-DIAGNOSTIC]", JSON.stringify(req.body)) and return 200 { ok: true }. No database storage, no complexity. We just need it in the server logs.
Wrap in try/catch. If the logging endpoint itself fails, return 200 anyway — this must never interfere with the user's experience.

Step 2 — Client-side: report errors from uploadSinglePhoto in client/src/pages/tech-submit.tsx
Add a small helper function near the top of the component (above uploadSinglePhoto):
function reportUploadError(details: Record<string, unknown>) {
  try {
    const token = getToken();
    fetch("/api/uploads/report-error", {
      method: "POST",
      headers: {
        "Content-Type": "application/json",
        ...(token ? { Authorization: `Bearer ${token}` } : {}),
      },
      body: JSON.stringify({
        ...details,
        userAgent: navigator.userAgent,
        timestamp: new Date().toISOString(),
      }),
    }).catch(() => {}); // fire-and-forget, never block the UI
  } catch {}
}
Now update uploadSinglePhoto to call reportUploadError at each failure point. Do NOT change any existing logic, return values, retry behavior, or timing. Just add the report call right before the existing failure handling:

When urlRes.ok is false (line ~116): Before throwing, call reportUploadError({ stage: "request-url", fileName: file.name, fileSize: file.size, fileType: file.type, errorMessage: "HTTP " + urlRes.status, attempt }).
In the XHR load handler when status is NOT 2xx (line ~127-128): Before resolving null, call reportUploadError({ stage: "xhr-upload", fileName: file.name, fileSize: file.size, fileType: file.type, xhrStatus: xhr.status, attempt }).
In the XHR error handler (line ~130): Before resolving null, call reportUploadError({ stage: "xhr-error", fileName: file.name, fileSize: file.size, fileType: file.type, errorMessage: "network error", attempt }).
In the XHR timeout handler (line ~131): Before resolving null, call reportUploadError({ stage: "xhr-timeout", fileName: file.name, fileSize: file.size, fileType: file.type, errorMessage: "timeout after 120s", attempt }).
