# REPLIT PROMPT — Add "Parts NLA" Technician Submission Type

We are adding a third request type to the technician submission form: **Parts No Longer Available (NLA)**. This is for when a technician looks up a part in TechHub and it shows as unavailable to order through normal channels. The tech submits the part info and photos so the VRS parts team can research sourcing.

This prompt covers ONLY the technician-facing submission changes and the minimal backend validation updates needed to accept the new type. Agent/admin dashboard changes will come in a separate prompt.

---

## PART 1 — SCHEMA UPDATE (shared/schema.ts)

Add a new column to the `submissions` table:

```
partNumbers: text("part_numbers"), // JSON string array, e.g. '["WPW10321304","W10321304"]'
```

Place it after the `estimateAmount` column. This is nullable — only used when `requestType === 'parts_nla'`.

Add a comment on the `requestType` column updating the allowed values:
```
requestType: text("request_type").notNull(), // 'authorization', 'infestation_non_accessible', 'parts_nla'
```

Run `db:push` after this change. No migration file needed — Drizzle push handles it.

---

## PART 2 — BACKEND VALIDATION (server/routes.ts)

### 2a. createSubmissionSchema (~line 472)

Update the requestType enum:
```ts
requestType: z.enum(["authorization", "infestation_non_accessible", "parts_nla"]),
```

Add a new optional field to the schema:
```ts
partNumbers: z.string().optional(), // JSON string of part number array
```

### 2b. Submission creation handler (POST /api/submissions)

In the `storage.createSubmission()` call, pass the new field:
```ts
partNumbers: parsed.data.partNumbers || null,
```

### 2c. isNonPartsRequest logic (~line 1161, inside the process route)

The existing logic is:
```ts
const isNonPartsRequest = submission.requestType !== "authorization";
```

This already handles `parts_nla` correctly — it will be treated as a non-parts-auth request (no RGC auth code required). **Leave this as-is for now.** The agent-side NLA resolution flow will be built in a later prompt with its own action handling.

### 3d. Add Part Numbers input section (conditional)

When `watchedRequestType === "parts_nla"`, show a new card section between the Issue Description and the Photo sections. This section does NOT appear for other request types.

**Label:** "Part Number(s) *" with a `HelpTooltip` that says "Enter each unavailable part number from TechHub. Add additional parts using the + button."

**Behavior:**
- Start with one empty text input field, placeholder: "Enter part number (e.g., WPW10321304)"
- Each input has a small red X button on the right to remove it (except when there's only one — the first field cannot be removed)
- Below the input(s), show a small "+ Add Another Part" button that adds a new empty input field
- Maximum 10 part number fields
- At least one part number is REQUIRED when requestType is `parts_nla` — validate on submit. Show a red validation message "At least one part number is required" if all part number fields are empty on submit.
- Part number inputs should be uppercase-forced (transform input to uppercase on change)

**State management:** Use a `useState<string[]>` initialized to `[""]`. When adding, push an empty string. When removing, filter by index. When changing, update by index.

**Layout example:**
```

│ Enter each unavailable part number from TechHub.        │

│ ┌──────────────────────────────────────────────┐ [X]   │
```

### 3e. Photo sections for NLA

When `requestType === "parts_nla"`, the photo sections should behave exactly the same as `requestType === "authorization"`:
- **Issue Photos** section (required, up to 15) — label and description stay the same. For NLA context, the tech is uploading photos of the failed/needed part.
- **Model, Serial & Estimate Screenshots** section (required, up to 5) — stays the same.

Both sections remain required (at least 1 photo each). No changes to photo validation, upload logic, or labels needed. This mimics the authorization flow exactly.

### 3f. onSubmit handler update

Before calling `mutation.mutate()`, if `requestType === "parts_nla"`, serialize the part numbers array:

```ts
if (form.getValues("requestType") === "parts_nla" && partNumbers.length > 0) {
  const filteredParts = partNumbers.filter(p => p.trim() !== "");
  if (filteredParts.length > 0) {
    payload.partNumbers = JSON.stringify(filteredParts);
  }
}
```

### 3g. Estimate amount field

The estimate amount input should remain visible for `parts_nla` (same as authorization). Leave existing conditional logic as-is.

### 3h. AI Enhance button

Leave the AI Enhance button available for `parts_nla` — the tech can still use it to improve their issue description. No changes needed.

### 3i. Submit button label

When `requestType === "parts_nla"`, change the submit button text from "Submit for Review" to "Submit NLA Request". The button remains disabled until all required fields are filled (same validation pattern).
