REPLIT PROMPT 3 — Real-Time Notifications + Auto-Assignment + Division Correction

We are implementing the final core system changes to the VRS Agent Core platform: real-time push notifications, automatic ticket routing to available agents, and the ability to correct a ticket's division mid-review. Please implement all of the following carefully and in order.

PART 1 — REAL-TIME WEBSOCKET NOTIFICATIONS
The ws package is already installed. Implement a persistent WebSocket server on the backend (server/index.ts or a dedicated server/websocket.ts) that:

Maintains a live connection for each authenticated agent session
Associates each WebSocket connection with the logged-in user's ID and role
Broadcasts targeted events to specific agents based on their assigned divisions
Stays running constantly — agents should never need to refresh their screen to receive updates

Notification trigger — New ticket in queue:
When a technician submits a new ticket, the server immediately identifies all online/available agents whose assigned divisions match the ticket's appliance type, then pushes a notification to each of those agents.
Notification format on the frontend (sound + visual popup together):

Sound: Play a distinct notification "ding" sound
Visual: Display a toast/popup notification in the top right corner of the agent dashboard that does not auto-dismiss for at least 8 seconds
Message format: "New [appliance type] ticket, [warranty company]"

Examples:

"New Refrigeration ticket, SPHW"
"New Laundry ticket, American Home Shield"
"New Cooking ticket, First American"

Use these exact warranty company label abbreviations in notifications:

Sears Protect → SPHW
Sears PA → Sears PA
Legacy Sears / Cinch → Cinch
American Home Shield → American Home Shield
First American → First American

Notifications should only fire for agents whose status is online — do not send notifications to agents who are offline or already working.
Also broadcast the following WebSocket events to keep all connected agent dashboards in sync without refreshing:

Ticket claimed by an agent → remove from other agents' queue views in real time
Ticket returned to queue (rejected or reassigned) → appear in matching agents' queue views in real time
Agent status changes → update the admin's live agent status panel in real time


PART 2 — AUTOMATIC TICKET ROUTING
When a new ticket is submitted, the backend should automatically determine which agents are eligible to receive it based on:

Agent's assigned divisions match the ticket's appliance type
Agent's current status is online (not offline or working)

The ticket enters the shared division queue and is immediately visible to all matching online agents. The first agent to click/accept it owns it (as established in Prompt 2 — status flips to pending, agent status flips to working).
If no qualified agents are online when a ticket is submitted:

Ticket sits in the queue in queued status
When an agent comes online (toggles from offline to online), immediately check the queue for any unassigned tickets matching their divisions
If matches exist, fire the notification sound and popup for those tickets


PART 3 — DIVISION CORRECTION ON TICKET
Any agent who has a ticket open (in pending status) should be able to correct the appliance type/division if the technician submitted it under the wrong category.
Add a small, clearly labeled "Correct Division" button or dropdown near the appliance type field on the ticket detail view.
When clicked:

Show a dropdown with the full division list:

Refrigeration
Laundry
Cooking
Dishwasher / Compactor
Microwave
HVAC
All Other


Agent selects the correct division and confirms

On confirmation:

Update the ticket's appliance type/division in the database
Record the correction in the ticket's history log (e.g., "Division corrected from Refrigeration to Laundry by [agent name]")
If the correcting agent is NOT assigned the new division:

Clear assignedTo, return ticket to queued status
Set agent's status back to online
Send a real-time WebSocket event to all agents assigned the correct division notifying them of the newly available ticket
The correcting agent's ticket view closes and they return to the queue


If the correcting agent IS assigned the new division:

They keep ownership of the ticket and continue processing it
No queue change needed




IMPORTANT NOTES:

WebSocket connections should gracefully handle disconnects and reconnects — if an agent's browser closes and reopens, their connection should re-establish cleanly on next load
Do not break any existing functionality: Snowflake sync, SHSAI integration, photo handling, SMS, resubmission logic, and all technician-facing pages remain untouched
The VRS_MASTER super_admin account is exempt from all routing and notification logic
Update replit.md to reflect the WebSocket architecture and auto-routing system
