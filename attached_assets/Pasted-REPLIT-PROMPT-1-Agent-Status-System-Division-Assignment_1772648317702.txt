REPLIT PROMPT 1 — Agent Status System + Division Assignment + Queue Filtering

We are making three interconnected changes to the VRS Agent Core platform. Please implement all of the following carefully and in order.

PART 1 — AGENT STATUS SYSTEM
Add a new agentStatus column to the users table in shared/schema.ts with three possible values: "online", "working", "offline". Default is "offline".
Run db:push to apply the schema change.
Update server/storage.ts and server/routes.ts to support:

PATCH /api/agent/status — allows an authenticated vrs_agent to set their own status to online or offline (they cannot manually set working)
PATCH /api/admin/users/:id/status — allows admin or super_admin to set any agent's status (including forcing them offline)
GET /api/admin/agent-status — returns a live list of all vrs_agents with their current name, LDAP ID, status, and assigned divisions. Admins use this to see who is online, working, or offline at any moment.

Status automation rules (enforce server-side and reflect on frontend):

When an agent accepts/opens a ticket → automatically set their status to working
When an agent closes/navigates away from a ticket, or finishes processing one → automatically set status back to online
Status only resets to online if the agent was previously working — do not change an offline agent's status automatically

On the agent dashboard (client/src/pages/agent-dashboard.tsx):

Add a prominent Online/Offline toggle — this should be large, immediately visible, and centrally placed (top of sidebar or top bar). Use a clear visual treatment: green with a filled circle icon for Online, gray with an empty circle for Offline. This is the agent's "clock-in" action every morning.
Login popup: Immediately after login, if the agent's status is offline, show a modal dialog with the message: "You're currently offline. Go online to start receiving tickets?" with two buttons: Go Online and Stay Offline.
Persistent banner: While the agent's status is offline, show a persistent yellow/amber banner at the top of the dashboard that reads: "You are offline and not receiving tickets. Click here to go online." Clicking the banner sets them online. The banner disappears only when they go online.
Status indicators visible on the agent's own dashboard:

🟢 Green dot = Online (available)
🟡 Yellow dot = Working (on a ticket)
⚫ Gray dot = Offline



On the admin dashboard (client/src/pages/admin-dashboard.tsx):

Add a new "Agent Status" section or tab that shows a live-updating table of all agents with columns: Name, Status (colored dot + label), Assigned Divisions, Last Seen.
Admin can click an agent row and force their status to Offline (soft-disable from queue without deactivating account).


PART 2 — DIVISION LIST + ASSIGNMENT
Update the division list everywhere in the application to use this exact set:

Refrigeration
Laundry
Cooking
Dishwasher / Compactor
Microwave
HVAC
All Other

This applies to the admin dashboard division assignment UI, the submission form appliance type options, and all queue filtering logic.
In the admin dashboard, the existing division assignment checkbox grid should reflect this updated list. Admins and super_admins can assign any combination of divisions to any agent.
No divisions = no tickets. If an agent has zero divisions assigned and is not an admin, they should see an empty queue with a message: "You have no divisions assigned. Please contact your administrator."
A generalist agent is simply one assigned all divisions — no special flag needed.

PART 3 — SKILLS-BASED QUEUE FILTERING (BUG FIX)
Fix the current bug where agents can see and interact with tickets outside their assigned divisions.
Rules to enforce on both backend and frontend:

An agent may only see tickets in the queue that match at least one of their assigned divisions
An agent may only click or accept tickets that match their divisions — any attempt to access a ticket outside their division should return a 403 error
Admins and super_admins are exempt and can see all tickets regardless of division
If an agent is assigned "All Other," they see tickets submitted under the All Other category

Update GET /api/submissions and GET /api/submissions/:id in server/routes.ts to strictly enforce division matching when the requesting user is a vrs_agent.

Do not break any existing functionality — the existing Stage 1/Stage 2 workflow, Snowflake sync, SHSAI integration, SMS, and photo handling should remain untouched.
