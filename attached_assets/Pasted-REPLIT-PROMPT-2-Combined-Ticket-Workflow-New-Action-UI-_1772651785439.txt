REPLIT PROMPT 2 — Combined Ticket Workflow + New Action UI + Auth Code Logic

We are overhauling the core ticket processing workflow in the VRS Agent Core platform. This replaces the existing two-stage system with a single unified ticket view. Please implement all of the following carefully and in order.

PART 1 — TICKET OWNERSHIP + PENDING STATUS
Add a new ticketStatus column to the submissions table in shared/schema.ts with the following possible values: "queued", "pending", "approved", "rejected", "invalid", "completed". Default is "queued".
Run db:push to apply the schema change.
Ownership rules to enforce:

When a ticket is submitted by a technician → status is queued
When an agent clicks/opens a ticket → status immediately changes to pending, the ticket is assigned to that agent, and their own status flips to working (per Prompt 1 logic)
A pending ticket is locked to the owning agent — no other agent can open or accept it
If an agent needs to go offline mid-ticket, they must first request reassignment (see Part 3 below) before their status can be set to offline
Once the agent sends a resolution (Approve / Reject / Invalid) → ticket moves to the appropriate final status
After a Reject → ticket returns to queued status, assignedTo is cleared, and it re-enters the general division queue for any qualified agent to pick up. It does NOT return to the original agent specifically, though they may pick it up again if they choose.
After technician resubmission of a rejected ticket → new submission enters the queue as queued, available to any qualified agent


PART 2 — SINGLE UNIFIED TICKET VIEW
Remove the separate Stage 1 and Stage 2 tabs/queues from the agent dashboard. Replace with:
Queue View:

A single queue showing all queued tickets available to the agent (division-filtered per Prompt 1)
A separate section showing the agent's own pending ticket(s) they are currently working
A completed section showing resolved tickets (approved, rejected, invalid)

Single Ticket Detail View:
When an agent opens a ticket, everything happens in one screen in this order:

Submission details — all technician-submitted info, photos, service order history (SHSAI), etc. as currently displayed
Action section — new UI described in Part 3 below
Auth code section — conditionally shown based on warranty company (described in Part 4 below)

There is no separate Stage 2 screen. The agent who approves a ticket also enters the auth code on the same screen before the ticket is considered complete.

PART 3 — NEW ACTION UI
Replace the existing Approve / Reject / Invalid buttons with the following new UI at the bottom of the ticket detail view:
Action Selection — Large Checkboxes:
Display three large, clearly labeled checkboxes (not radio buttons — they should be easy to check and uncheck freely):

☐ Approve
☐ Reject
☐ Invalid

Only one can be selected at a time — selecting one automatically deselects the others. No action is taken by selecting a checkbox alone.
Reason Suggestions (contextual, not required):

If Reject is checked → display a list of suggestion checkboxes below (multiple can be selected, none are required):

No pictures sent
Picture doesn't show voltage
Need more pictures
Incomplete pictures
Video blurry
(Additional options to be added later)


If Invalid is checked → display a brief note: "This ticket will be marked as a non-VRS issue." No sub-options needed.
If Approve is checked → show the auth code section (see Part 4)

Freehand Text Box:
Always visible regardless of which action is selected. Labeled: "Additional message to technician (optional)". Not required. Gives the agent free-form space to customize their message.
Process Button:

Large, clearly distinct button in the bottom right corner of the ticket
Labeled "Process & Send"
Disabled (grayed out) until at least one action checkbox is selected
When clicked → triggers a confirmation popup

Confirmation Popup:

Shows a summary of what is about to be sent:

Selected action (Approve / Reject / Invalid)
Selected reasons (if any)
Freehand message (if any)
For Approve: preview of the message that will be sent to the technician (see below)


Two buttons: Confirm & Send and Cancel

Approval message sent to technician on confirm:

"Your submission has been approved. Please do not leave the site. You may begin cleaning up your workspace and explaining the process to the customer. You will receive your authorization or instructions shortly."


PART 4 — AUTH CODE LOGIC BY WARRANTY COMPANY
After an agent selects Approve and before they can hit Process & Send, display the appropriate auth code input(s) based on the ticket's warranty company. This appears inline on the same ticket screen, below the action section.
Rules by warranty company:

Sears Protect → Auto-populate the RGC code from the daily code already stored in the admin panel. Agent does not need to enter anything. Display it as read-only so they can confirm it.
Sears PA → Same as Sears Protect — auto-populate RGC code, read-only display.
Legacy Sears / Cinch → Same as Sears Protect — auto-populate RGC code, read-only display.
American Home Shield → Show two fields:

RGC Code (auto-populated, read-only)
AHS Authorization Code (agent manually enters the code they received by calling AHS)


First American → Same as American Home Shield:

RGC Code (auto-populated, read-only)
First American Authorization Code (agent manually enters)



No auth code required for tickets that are non-standard requests (e.g., multiple-attempt replacement review). In these cases the Approve flow skips the auth code section entirely and goes straight to the confirmation popup. Determine this based on the submission's requestType field — if it is flagged as a replacement review or similar non-parts request, skip auth code entry.
All auth codes entered should be stored on the submission record and included in the SMS sent to the technician upon completion.

PART 5 — MID-TICKET REASSIGNMENT REQUEST
If an agent needs to go offline while they have a pending ticket:

They cannot simply toggle offline — the system should prompt: "You have an open ticket. Please request reassignment before going offline."
Show a Request Reassignment button on their pending ticket
When clicked → show a notes field: "Add any notes for the next agent (optional)"
On confirm → save their notes to the ticket, clear assignedTo, return ticket to queued status in the division queue, then allow the agent to go offline
All activity and notes from the original agent are preserved in the ticket history
